## Becoming a Maintainer

Maintainers are added based on sustained, high-quality contributions and deep understanding of the project. The process:

1. **Consistent Contributions**: Regular, high-quality PRs over 6+ months
2. **Domain Expertise**: Deep knowledge of one or more subsystems
3. **Community Engagement**: Active in issues, discussions, and reviews
4. **OpenSpec Alignment**: Understanding of spec-driven development workflow
5. **Nomination**: Existing maintainer nominates the contributor
6. **Consensus**: Current maintainers reach consensus (2/3 majority)

### Expected Commitments
- Review PRs within 1 week
- Respond to critical issues within 48 hours
- Participate in monthly maintainer meetings
- Follow OpenSpec workflow for all changes
- Maintain assigned subsystems

## Decision Making

### Types of Decisions

| Decision Type | Process | Approvals Required |
|---------------|---------|-------------------|
| Bug fixes | Standard PR review | 1 maintainer |
| Minor features | PR + OpenSpec change | 1 maintainer |
| Major features | PR + OpenSpec proposal | 2 maintainers |
| Breaking changes | RFC + OpenSpec proposal | 2/3 majority |
| Governance changes | RFC + discussion | Consensus |

### Conflict Resolution
1. Discussion in PR/issue
2. Escalate to maintainer sync meeting
3. Lead maintainer makes final decision if no consensus
